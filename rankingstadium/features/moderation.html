モデレーション機能（ブロック・通報）

App Store ガイドライン 1.2 / 5.1.1 対応 — ユーザーが有害コンテンツ・投稿者を自衛できる仕組み

v1.0 機能ID: RS-FEAT-MOD

概要

App Store Guideline 1.2（UGCアプリはモデレーション手段を提供する義務）および 5.1.1（プライバシー保護）への対応として、以下の2機能を実装した。

  • ブロック機能 — 特定ユーザーをブロックし、フィード全体からその投稿を除外する
  • 通報機能 — ランキングやユーザーを8種類の理由で通報し、運営が確認できる仕組みを提供する

関連画面

  • ユーザープロフィール（ブロック導線: ヘッダーメニュー）
  • ランキング詳細（ブロック・通報導線: ツールバー「…」メニュー）
  • ホーム（ブロックユーザーの投稿除外が適用される）
  • 検索（ブロックユーザーの投稿除外が適用される）
  • マイページ（フォロー中フィードへのブロック除外適用）

1. ブロック機能

BlockService API

メソッド説明
block(targetUserId: UUID)ブロック実行。blocks テーブルに INSERT。同時にフォロー関係を双方向で解除する。
unblock(targetUserId: UUID)ブロック解除。blocks テーブルから DELETE。
isBlocked(targetUserId: UUID) async -> Bool対象ユーザーをブロック中か判定。
fetchBlockedUserIds() async throws -> [UUID]自分がブロックしたユーザーIDの一覧を取得。RetryHelper で保護。

フィード除外の仕組み

BlockService.fetchBlockedUserIds() で取得したID群を PostgREST の否定IN（creator_id=not.in.(id1,id2,...)）でクエリに挿入する excludingBlockedCreators ヘルパーを介して、以下の全フィードに適用する。
フィード適用箇所
ホーム — 新着 / 人気 / 終了間近 / 終了後全タブ共通で excludingBlockedCreators を挿入
検索 — 投票中 / 終了後検索クエリ構築時に挿入
マイページ — フォロー中フォロー取得後のフィルタで挿入

ブロック処理フロー

sequenceDiagram participant U as User participant V as ProfileView / RankingDetailView participant VM as ViewModel participant BS as BlockService participant FS as FollowService participant DB as Supabase U->>V: 「…」メニュー → 「このユーザーをブロック」 V->>VM: block(targetUserId) VM->>BS: block(targetUserId) BS->>DB: INSERT blocks (blocker_id, blocked_id) BS->>FS: unfollow(targetUserId) / removeFollower(targetUserId) FS->>DB: DELETE follows (双方向) DB-->>BS: 成功 BS-->>VM: 完了 VM-->>V: UI更新（ブロック済み表示） V-->>U: 確認トースト表示

ブロック導線

画面導線補足
ProfileViewヘッダー右の「…」ボタン → 「ブロック」自分のプロフィールでは表示しない
RankingDetailViewツールバー「…」メニュー → 「この投稿者をブロック」匿名ランキングでも creator_id は内部的に存在するため到達可能

ビジネスルール（ブロック）

  • 自分自身のブロック不可（DB CHECK 制約 + UI 側でも防止）
  • ブロック時はフォロー関係（自分→相手 / 相手→自分 の両方）を自動解除
  • ブロックIDの取得は RetryHelper で最大3回リトライ保護
  • ブロック後も相手のランキングURLを直接開いた場合は表示する（フィードから除外するが閲覧まで制限しない）
  • ブロックされた側には通知を送らない

2. 通報機能

ReportReason 一覧

enum ケース表示ラベル
discrimination差別・ヘイトスピーチ
violence暴力・脅迫
self_harm自傷・自殺に関する内容
sexual性的なコンテンツ
illegal違法行為・詐欺
personal_info個人情報の無断掲載
spamスパム・繰り返し投稿
otherその他

通報導線

通報の導線は RankingDetailView のツールバー「…」メニューに集約する。 投票前（VoteView / ThreePointVoteView / 結果ロック画面）でも同メニューから通報可能。
画面・状態通報可否
RankingDetailView（投票前）可
VoteView / ThreePointVoteView可（ツールバー経由）
結果ロック画面（ResultLockedView）可
結果画面（ResultView）可

通報処理フロー

sequenceDiagram participant U as User participant V as RankingDetailView participant VM as ReportViewModel participant DB as Supabase reports U->>V: 「…」→ 「通報する」 V->>U: 通報理由シート表示（8種類） U->>V: 理由を選択 V->>VM: submitReport(rankingId, reason) VM->>DB: INSERT reports (reporter_id, ranking_id, reason) DB-->>VM: 成功 VM-->>V: 完了 V-->>U: 「通報しました」トースト

ビジネスルール（通報）

  • 同一ユーザーが同一ランキングを複数回通報することは可能（理由が異なる場合を考慮し制限しない）
  • 通報後の処理（非表示化・削除）は運営が手動で対応する
  • 通報者に対して結果フィードバックはしない
  • 通報データは reports テーブルに保存し、isResolved フラグで解決済みを管理

外部連携

連携先用途備考
Supabase blocks テーブルブロック関係の保存・参照RLS で当事者のみ可視
Supabase reports テーブル通報データの保存既存テーブル
FollowServiceブロック時のフォロー関係解除双方向 unfollow
RetryHelperfetchBlockedUserIds の通信失敗保護最大3回リトライ

エラー処理

発生条件対応
blocks テーブルへの INSERT 失敗（通信エラー）RetryHelper でリトライ。失敗時はエラーアラート表示
UNIQUE 制約違反（すでにブロック済み）「すでにブロックしています」メッセージ表示（UI側では通常起きない）
CHECK 制約違反（自己ブロック）UI 側で防止しているため通常到達しない
fetchBlockedUserIds 失敗空リストにフォールバック（フィード除外が動かなくなるため、次回起動時に再試行）
通報 INSERT 失敗エラーアラート表示。リトライは手動（再度通報操作）

実装メモ

  • BlockService — block / unblock / isBlocked / fetchBlockedUserIds 実装済み
  • excludingBlockedCreators ヘルパー — ホーム全タブ・検索・マイページフォロー中フィードに適用済み
  • ブロック導線 — ProfileView ヘッダー + RankingDetailView ツールバー「…」メニュー
  • 匿名ランキングでも creator_id は内部的に保持しており「この投稿者をブロック」に到達可能
  • 通報導線 — RankingDetailView ツールバー「…」メニューに集約（投票前/後/ロック画面すべてで通報可）
  • ReportReason 8種実装済み
  • ブロック一覧画面（自分がブロックしたユーザーを確認・解除）は v1.1 以降で検討
  • reports テーブルの運営管理ダッシュボード（Supabase Studio で対応）は別途整備
