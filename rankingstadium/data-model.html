データモデル

Supabase（PostgreSQL）テーブル ⇔ Swift Codable モデルの対応

本アプリは SwiftDataではなく Supabase をバックエンドに採用。各モデルは struct + Codable で定義し、Supabase クライアント経由で読み書き。RLS（Row Level Security）でユーザー権限を制御。

主要モデル

AppUser

struct AppUser: Codable, Identifiable {
    let id: UUID
    var displayName: String
    var isPremium: Bool
    var isComp: Bool              // 招待プレミアム枠フラグ（CodingKey: is_comp）
    var points: Int
    var loginStreak: Int
    var freeCreationRemaining: Int
    // ... 他プロフィール項目

    // StoreKit課金または招待枠のどちらかでプレミアム機能を解放する集約プロパティ
    var hasPremiumAccess: Bool { isPremium || isComp }

    enum CodingKeys: String, CodingKey {
        case id, displayName = "display_name", isPremium = "is_premium"
        case isComp = "is_comp", points, loginStreak = "login_streak"
        case freeCreationRemaining = "free_creation_remaining"
    }

    init(from decoder: Decoder) throws {
        let c = try decoder.container(keyedBy: CodingKeys.self)
        id = try c.decode(UUID.self, forKey: .id)
        displayName = try c.decode(String.self, forKey: .displayName)
        isPremium = try c.decodeIfPresent(Bool.self, forKey: .isPremium) ?? false
        isComp = try c.decodeIfPresent(Bool.self, forKey: .isComp) ?? false  // 旧クライアント互換
        points = try c.decode(Int.self, forKey: .points)
        loginStreak = try c.decodeIfPresent(Int.self, forKey: .loginStreak) ?? 0
        freeCreationRemaining = try c.decodeIfPresent(Int.self, forKey: .freeCreationRemaining) ?? 0
    }
}

users テーブル — is_comp カラム

マイグレーション: supabase/migrations/028_add_is_comp.sql
カラム型説明
is_comp BOOLEAN NOT NULL DEFAULT FALSE 招待プレミアム枠フラグ。運営が Supabase SQL Editor（service_role）から手動付与・取り消しを行う。StoreKit 課金とは別系統で、syncPremiumStatusToSupabase() による起動時同期では上書きされない（恒久付与）。アプリ側は読み取り専用。

列レベルセキュリティ

028_add_is_comp.sql にて authenticated / anon ロールの is_comp への UPDATE 権限を剥奪する。他カラムへの UPDATE 権限は維持（REVOKE 全体 → is_comp を除く全列を GRANT）。

アプリ側は is_comp を書き込まない: syncPremiumStatusToSupabase() の UPDATE 対象カラムから is_comp を意図的に除外していること。誤って含めると RLS により 403 エラーとなる。

整合性に関する注意

非アトミック操作: RankingService.createRanking() と VoteService.castVote() は複数の Supabase RPC/INSERT をシーケンシャル実行する。途中失敗時の整合性を保証するため、本番運用では Edge Function 化を検討すること。
二重付与防止: PointService.claimLoginBonus() はクライアント側 lastLoginDate + DB point_transactions の二層チェックで日次ボーナスの二重付与を防いでいる。
users.last_login_date は DATE 型・クライアントから "yyyy-MM-dd" 文字列で送信: users.last_login_date は Supabase の DATE 型カラム。Swift Date 型のまま送ると Supabase が ISO8601 タイムスタンプを UTC で日付化するため、JST 端末では JST 5/30 00:00 が UTC 5/29 15:00 → "2026-05-29" として保存される（前日扱い）。これを防ぐため、クライアントは LoginBonusUpdate.lastLoginDate を ローカルTZ基準の "yyyy-MM-dd" 文字列（pointDateOnlyFormatter で生成）で送信する。

blocks テーブル

App Store ガイドライン 1.2（ユーザー生成コンテンツのモデレーション義務）および 5.1.1（プライバシー保護）対応として追加。 マイグレーション: supabase/migrations/027_blocks.sql

カラム定義

カラム型説明
idUUID PKブロックレコード識別子（auto-generated）
blocker_idUUID, FK → users(id) ON DELETE CASCADEブロックした側の内部ユーザーID
blocked_idUUID, FK → users(id) ON DELETE CASCADEブロックされた側の内部ユーザーID
created_atTIMESTAMPTZブロック日時
blocker_id / blocked_id は内部 users.id（auth_id ではない）。アカウント削除時は users への ON DELETE CASCADE で blocks レコードも自動削除される。

制約

制約内容
UNIQUE(blocker_id, blocked_id) — 同一ペアの二重ブロック防止
CHECKblocker_id <> blocked_id — 自己ブロック防止

RLS ポリシー

ポリシー名操作条件
blocks_selectSELECTblocker_id = auth.uid() OR blocked_id = auth.uid()（当事者のみ可視）
blocks_insertINSERTblocker_id = auth.uid()（自分がブロックする場合のみ）
blocks_deleteDELETEblocker_id = auth.uid()（自分のブロックのみ解除可）

削除順序

設計上の前提（FK 制約）
users.auth_id → auth.users(id) は ON DELETE CASCADE 設定済みのため、auth.users を削除すれば public.users も自動削除される。
しかし rankings.creator_id / votes / choice_requests / reports / point_transactions / favorites / unlocked_rankings / device_fingerprints の user・ranking 参照は CASCADE 未設定。そのため auth.users をいきなり削除すると FK 制約違反になる。よって RPC 内で以下の順に明示削除する。
順序削除対象削除キー補足
1 自分が作成したランキングに紐づく子レコード（他ユーザー分含む） ranking_id 参照 votes / choice_requests / reports / favorites / unlocked_rankings / point_transactions / notifications。choices / quizzes は rankings への CASCADE で同時削除されるため明示不要。
2 自分自身が他ランキングに対して作成したデータ user_id / requester_id / reporter_id 参照 votes / choice_requests / reports / favorites / unlocked_rankings / point_transactions / device_fingerprints
3 自分が作成した rankings — choices / quizzes は CASCADE で同時削除
4 auth.users — public.users が CASCADE で自動削除。follows / notification_settings / notifications(user_id) も CASCADE。feedbacks.user_id は SET NULL で匿名化。

iOS 側実装

  • AuthViewModel.deleteAccount() が RPC を呼び出す。
  • 成功後はローカルセッションのみクリア（signOut(scope: .local)）して authState = .unauthenticated に遷移。
  • 設定画面で確認アラート・削除されるデータの説明・失敗時アラートを表示。
未対応事項: Sign in with Apple のリフレッシュトークン revoke（Apple REST API）はサーバー（Edge Function）が必要なため本バージョンでは未実装。審査で revoke まで求められた場合に別途 Edge Function を追加する。
